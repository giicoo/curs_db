# Задача: реализовать backend + frontend для АИС «Интернет-маркетплейс»

Это курсовая работа по дисциплине «Базы данных». **База данных и таблицы уже созданы и заполнены тестовыми данными** — не пересоздавай схему, не меняй DDL, не запускай миграции. Твоя задача — построить REST API и SPA-клиент поверх существующей структуры.

## 1. Существующая инфраструктура (не трогать)

В корне проекта уже лежит `docker-compose.yml`, поднимающий два контейнера:

```yaml
services:
  postgres:
    image: postgres:16
    container_name: pg_marketplace
    environment:
      POSTGRES_DB: marketplace
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
    ports:
      - "5432:5432"
    volumes:
      - pg_data:/var/lib/postgresql/data
      - ./db/init:/docker-entrypoint-initdb.d:ro

  adminer:
    image: adminer:latest
    ports:
      - "8080:8080"
```

Подключение к БД из приложения: `postgresql://postgres:postgres@localhost:5432/marketplace` (или `@postgres:5432` если backend тоже будет в docker-compose — в этом случае добавь сервис `backend` в тот же `docker-compose.yml`, подключи его к сети `marketplace_net`, не меняя существующие сервисы).

Таблицы создаются скриптами `db/init/01_create_tables.sql` и `db/init/02_seed_data.sql`, которые выполняются автоматически при первом поднятии тома — **не изменяй и не дублируй их логику в коде приложения**.

## 2. Схема базы данных (источник истины)

9 таблиц, все связи 1:1 / 1:М, без М:М (Заказ↔Товар разрешён через order_items с составным PK):

```sql
users (
    id SERIAL PRIMARY KEY,
    login VARCHAR(50) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE,
    registered_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
)

customers (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL UNIQUE REFERENCES users(id),
    last_name VARCHAR(100) NOT NULL,
    first_name VARCHAR(100) NOT NULL,
    phone VARCHAR(20),
    delivery_address VARCHAR(255)
)

sellers (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL UNIQUE REFERENCES users(id),
    name VARCHAR(150) NOT NULL UNIQUE,
    description TEXT,
    registered_at DATE NOT NULL DEFAULT CURRENT_DATE
)

product_types (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL UNIQUE
)

products (
    id SERIAL PRIMARY KEY,
    seller_id INTEGER NOT NULL REFERENCES sellers(id),
    type_id INTEGER NOT NULL REFERENCES product_types(id),
    name VARCHAR(255) NOT NULL,
    price NUMERIC(10,2) NOT NULL,
    stock_quantity INTEGER NOT NULL DEFAULT 0
)

order_statuses (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) NOT NULL UNIQUE,
    sort_order INTEGER NOT NULL,
    is_final BOOLEAN NOT NULL DEFAULT FALSE
)
-- значения: Оформлен(1), Собран(2), В доставке(3), Доставлен(4, final), Отменён(5, final)

orders (
    id SERIAL PRIMARY KEY,
    customer_id INTEGER NOT NULL REFERENCES customers(id),
    status_id INTEGER NOT NULL REFERENCES order_statuses(id),
    order_date TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    delivery_address VARCHAR(255) NOT NULL
)

order_items (
    order_id INTEGER NOT NULL REFERENCES orders(id),
    product_id INTEGER NOT NULL REFERENCES products(id),
    quantity INTEGER NOT NULL,
    price_at_order NUMERIC(10,2) NOT NULL,
    PRIMARY KEY (order_id, product_id)
)

reviews (
    id SERIAL PRIMARY KEY,
    customer_id INTEGER NOT NULL REFERENCES customers(id),
    product_id INTEGER NOT NULL REFERENCES products(id),
    rating SMALLINT NOT NULL CHECK (rating BETWEEN 1 AND 5),
    comment_text TEXT,
    created_at DATE NOT NULL DEFAULT CURRENT_DATE
)
```

**Тестовые пользователи уже в базе**: `user1`…`user300` — покупатели, `user301`…`user350` — продавцы, у всех пароль `Test1234!` (хеш Argon2id уже корректно посчитан и лежит в `password_hash`).

**Роль пользователя определяется не отдельным полем**, а тем, в какой из таблиц (`customers` или `sellers`) есть запись с данным `user_id`. При реализации auth-сервиса при логине нужно проверить обе таблицы и вернуть роль в составе JWT.

## 3. Технологический стек

**Backend:**
- Python 3.12, FastAPI
- `asyncpg` для подключения к PostgreSQL (или `psycopg` async — на твой выбор, главное консистентно)
- Pydantic v2 для схем запросов/ответов
- `argon2-cffi` (`argon2.PasswordHasher`) для проверки и хеширования паролей — **обязательно Argon2id**, не bcrypt
- `python-jose` или `PyJWT` для выдачи и проверки JWT-токенов (payload: `user_id`, `role` — `customer` или `seller`)
- Структура: роутеры по доменам (`auth`, `catalog`, `products`, `orders`, `reviews`, `reports`), отдельный слой доступа к БД (репозитории/queries), DI через `Depends` для текущего пользователя и его роли

**Frontend:**
- Vue 3, Composition API, `<script setup>`
- Vue Router 4 — с защищёнными маршрутами по ролям (`meta: { role: 'customer' | 'seller' }`)
- Pinia — хранение JWT-токена, текущего пользователя/роли, состояния корзины
- Axios — HTTP-клиент с interceptor, добавляющим `Authorization: Bearer <token>` ко всем запросам после логина
- Chart.js — для графиков в разделе аналитики

## 4. Функциональные требования

### 4.1 Аутентификация (`/auth`)

- `POST /auth/register` — регистрация. Принимает `login`, `password`, `email`, `role` (`customer`/`seller`) и поля профиля (для customer: ФИО, телефон, адрес; для seller: название магазина, описание). Транзакционно создаёт запись в `users`, затем в `customers` или `sellers`.
- `POST /auth/login` — принимает `login`, `password`. Ищет пользователя, проверяет пароль через `PasswordHasher().verify()`, определяет роль (наличие записи в customers/sellers), возвращает JWT.
- `GET /auth/me` — текущий пользователь по токену (id, login, email, role, данные профиля).

### 4.2 Каталог и товары (`/catalog`, `/products`)

- `GET /catalog` — публичный, без авторизации. Фильтры: `type_id`, `seller_id`, `min_price`, `max_price`, `search` (по названию), пагинация. Возвращает товары со средним рейтингом (подзапрос/JOIN к `reviews`).
- `GET /products/{id}` — карточка товара с отзывами.
- `POST /products` — только роль `seller`. Создаёт товар, `seller_id` берётся из токена, не из тела запроса.
- `PUT /products/{id}` — только владелец товара (проверка `seller_id` через токен).
- `DELETE /products/{id}` — только владелец.
- `GET /products/low-stock?threshold=10` — только seller, список своих товаров с остатком ниже порога (перспективная задача из ТЗ).
- `GET /product-types` — справочник для фильтров и форм.

### 4.3 Заказы (`/orders`)

- `POST /orders` — только `customer`. Тело: `delivery_address`, список позиций `[{product_id, quantity}]`. В одной транзакции: создать `orders` со статусом «Оформлен» (id=1), вставить все `order_items` с `price_at_order = products.price` на момент заказа, **уменьшить `stock_quantity`** у соответствующих товаров. Если остатка недостаточно — откатить транзакцию и вернуть ошибку.
- `GET /orders` — для customer: свои заказы; для seller: заказы, содержащие хотя бы один его товар (через JOIN с `order_items`/`products`).
- `GET /orders/{id}` — детали заказа с позициями.
- `PATCH /orders/{id}/status` — только seller, только для заказов со своими товарами. Принимает `status_id`. **Проверять, что переход допустим** — новый `sort_order` должен быть больше текущего (нельзя откатывать статус назад, кроме явной отмены).
- `GET /order-statuses` — справочник.

### 4.4 Отзывы (`/reviews`)

- `POST /reviews` — только customer. Принимает `product_id`, `rating` (1–5), `comment_text`. **Бизнес-правило**: разрешить отзыв только если у покупателя есть заказ со статусом «Доставлен» (`is_final = true`), включающий этот товар — проверить через JOIN orders/order_items.
- `GET /reviews?product_id=` — список отзывов товара, публичный эндпоинт.

### 4.5 Аналитика (`/reports`) — соответствует прикладным задачам из раздела 2 курсовой

- `GET /reports/sales?from=&to=&group_by=product|type|seller` — сумма и количество заказов за период.
- `GET /reports/top-products?limit=10` — топ товаров/продавцов по выручке.
- `GET /reports/order-status-distribution` — распределение текущих заказов по статусам (для круговой диаграммы).
- `GET /reports/customer-activity` — топ покупателей по сумме/количеству заказов за период.
- `GET /reports/review-summary?product_id=|seller_id=` — количество отзывов и средний рейтинг.

Реализуй эти эндпоинты через агрегирующие SQL-запросы (GROUP BY, агрегатные функции) — не вытягивай все строки в Python для агрегации на стороне приложения.

### 4.6 Разграничение доступа

Используй FastAPI `Depends` для получения текущего пользователя из JWT и отдельные зависимости `require_customer` / `require_seller`, которые возвращают `403 Forbidden`, если роль не совпадает. Для эндпоинтов с проверкой владения ресурсом (товар, заказ) — сверяй `seller_id`/`customer_id` из БД с тем, что в токене, а не доверяй ID, переданным в теле запроса.

## 5. Frontend — структура страниц

- `/login`, `/register` — формы входа и регистрации с выбором роли
- `/catalog` — каталог с фильтрами (доступен без авторизации)
- `/product/:id` — карточка товара + отзывы + форма добавления отзыва (если разрешено)
- `/cart` — корзина (состояние в Pinia, не персистится между сессиями)
- `/orders` — мои заказы (для customer) — список со статусами
- `/seller/products` — мои товары (CRUD) — только для seller
- `/seller/orders` — заказы с моими товарами, смена статуса — только для seller
- `/seller/low-stock` — товары на грани остатка — только для seller
- `/reports` — аналитика с графиками Chart.js — доступна обеим ролям, но с разным набором фильтров (seller видит только свои данные, если не предусмотрена отдельная админ-роль)

Защити маршруты `/seller/*` через `router.beforeEach`, проверяя роль из Pinia-стора; при попытке доступа без нужной роли — редирект на `/catalog`.

## 7. Что важно НЕ делать

- Не пересоздавай и не изменяй структуру таблиц — она финальна и описана в разделе 4 пояснительной записки курсовой.
- Не переходи на bcrypt/scrypt — обязательно Argon2id, иначе существующие тестовые пользователи (`user1`…`user350`, пароль `Test1234!`) не смогут войти.
- Не делай поле "роль" отдельной колонкой в `users` — роль определяется наличием записи в `customers`/`sellers`, это сознательное архитектурное решение (единая точка регистрации + 1:1 связи).
- Не агрегируй отчёты в Python — используй SQL (GROUP BY, агрегатные функции, при необходимости — представления или функции PostgreSQL).

## 8. Definition of done

- `docker compose up -d` поднимает postgres + adminer + backend + frontend одной командой.
- Можно зарегистрироваться как customer и как seller, либо войти тестовым пользователем (`user1`/`Test1234!` — customer, `user301`/`Test1234!` — seller).
- Seller может добавить товар, увидеть его в каталоге под своей учётной записью.
- Customer может найти товар в каталоге, оформить заказ, увидеть его в «Моих заказах».
- Seller видит входящий заказ, может сменить статус по порядку (сообразно `sort_order`).
- После статуса «Доставлен» customer может оставить отзыв на товар из этого заказа.
- Раздел отчётов отображает хотя бы 2-3 графика с реальными агрегированными данными из БД.
